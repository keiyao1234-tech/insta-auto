# COST_REPORT — 無料運用のための前提・限界・代替策

本システムは **完全無料・放置運用** を目標に設計している。以下に、各コンポーネントの無料条件と上限、課金が発生し得るポイントとその理由、無料を維持する代替策、そして「無料の壁」への対処方針をまとめる。

---

## 1. 各コンポーネントの無料条件と具体的上限

| コンポーネント | 役割 | 無料条件・上限 | 本システムの想定使用量 | 余裕度 |
|---|---|---|---|---|
| **Instagram Platform API** | 投稿・Insights・(P2)DM | プラットフォーム利用料 **$0**。公開上限 **25投稿/24h**、API **200コール/h/アカウント**、ハッシュタグ検索 **30ユニーク/7日** | 1日 数投稿・数十コール | ◎ 余裕 |
| **GitHub Actions（公開リポ）** | スケジューラ/実行環境 | **公開リポは実行時間 無制限・無料**（private は 2,000分/月） | 1ジョブ 1〜3分 × 日数回 | ◎ 公開なら無制限 |
| **GitHub（リポ/raw配信）** | コード・メディア・状態の保管と公開URL | 公開リポ無料。1ファイル **≤100MB**（推奨50MB）、リポ全体 推奨 **≤1GB**、raw帯域はソフトリミット（個人利用域では問題なし） | 画像/短尺動画 数十〜百MB規模 | ○ 動画量に注意 |
| **Discord Webhook** | 失敗アラート/(P2)DB通知 | 無料。レート上限 約 **30メッセージ/分** | 数通/日 | ◎ 余裕 |
| **Cloudflare Workers（Phase 2）** | DM受信 webhook | 無料プラン **100,000リクエスト/日**、CPU 10ms/req | DM受信のみ（極少） | ◎ 余裕 |
| **トークンリフレッシュ** | 60日失効回避 | API無料。`gh secret set` も無料 | 週1回 | ◎ |

**結論: 通常の個人サロン運用なら、どのコンポーネントも無料枠に到達しない。**

## 2. 課金が発生し得るポイントと理由 → 無料維持の代替策

### ① LLM によるキャプション/文章自動生成（既定 OFF）
- **課金理由**: Anthropic/OpenAI 等の LLM API はトークン課金。`config.yaml` `caption.llm.enabled: true` にすると発生し得る。
- **代替策（無料維持）**: 既定の **テンプレート生成**（`config.yaml` `caption.templates` + `hashtags.yaml`）を使う。これで文面もハッシュタグも完全無料。バリエーションはテンプレ追加で増やせる。
- **本システムの既定**: `llm.enabled: false`（= 無料）。

### ② リアルタイムの流行ハッシュタグ大量取得
- **課金/不可理由**: 公式ハッシュタグ検索は **30ユニーク/7日** が上限で、流行タグの大量・リアルタイム取得は公式無料では不可能。第三者の有料トレンドAPIは課金。スクレイピングは**規約違反・BANリスク**のため不採用。
- **代替策（無料維持）**: **キュレーション済みプール**（`hashtags.yaml`）を人手でメンテし、**多腕バンディット**が Insights 実績から「効くセット」を自動強化。流行は月1回など低頻度で手動更新すれば、検索上限内に十分収まる。

### ③ 常時稼働の Webhook（Phase 2: DM→Discord）
- **課金理由**: 24/7 公開エンドポイントを VPS 等で常時稼働させると課金。
- **代替策（無料維持）**: **Cloudflare Workers 無料枠（10万req/日）** を使う。サーバーレスで常時稼働・無料。`webhook/` に実装。

### ④ private リポジトリに切り替えた場合の Actions 分数
- **課金理由**: private リポは Actions が **2,000分/月** までで、超過は従量課金。
- **代替策（無料維持）**: (a) **公開リポのまま運用**（無制限）。(b) どうしてもコード/metricsを隠したい場合は「**コード・状態=private リポ（軽量ジョブで2,000分内に収める）＋ メディアのみ public リポ**」に分割（下記「プライバシー」参照）。1ジョブ数分 × 日数回なら private でも2,000分に収まる。

### ⑤ メディア量の増大（GitHub 容量/帯域）
- **課金理由**: 動画を大量に貯めるとリポ肥大。LFSの無料枠（1GBストレージ/1GB帯域・月）は小さく超過課金。
- **代替策（無料維持）**: **LFSを使わない**（そもそもraw配信に必要）。古い投稿済みメディアは定期的に削除/別保管。1ファイル50MB目安に圧縮。さらに必要なら **Cloudflare R2 無料10GB** へ移行（`media.raw_url` 相当を差し替え）。

## 3.「無料の壁」最有力候補と対処方針（まとめ）

| 無料の壁 候補 | なぜ壁になるか | 本システムの対処（=無料維持） |
|---|---|---|
| **リアルタイム流行タグ取得** | 公式は30/7日上限・有料API/スクレイピングは課金orBAN | キュレーションプール＋バンディット学習で代替（②） |
| **常時稼働 webhook** | 常時起動サーバ＝課金 | Cloudflare Workers 無料枠（③） |
| **LLM 文章生成** | トークン従量課金 | テンプレ生成を既定化（①） |
| **動画ホスティング/帯域** | 大容量・LFS課金 | 非LFS・圧縮・削除・必要時R2無料枠（⑤） |
| **private運用時のCI分数** | 2,000分/月上限 | 公開運用 or 軽量ジョブ＋メディア分離（④） |

## 4. プライバシーに関する補足（無料は維持したまま）
公開リポにすると **コード・`state/metrics.csv`（リーチ等の数値）も公開**される。美容室では影響は軽微だが、隠したい場合の無料構成:

- **コード＋状態 = private リポ**（GitHub Actions 2,000分/月以内：1ジョブ数分なので十分収まる）
- **メディア = 別の public リポ**（raw URL 配信用。`GITHUB_REPO`/`raw_base_url()` をそのメディアリポに向ける）
- もしくは **Cloudflare R2（無料10GB）** にメディアを置き、ランダムキーのURLで実質非公開配信。

いずれも **追加費用 $0** で実現可能。
